## 一、项目概述

### 一·一 业务背景

本系统面向 HR 与候选人的招聘场景，核心围绕「职位发布 → 候选人投递 → 面试状态跟踪」三条主线。HR 发布职位并管理候选人流程，候选人在职位广场浏览并投递简历。系统以看板形式直观展示招聘漏斗各阶段的人才分布。

### 一·二 核心业务闭环

```
HR 发布职位 → 候选人浏览广场 → 投递简历 → HR 筛选/面试/录用
                                    ↓
                              招聘漏斗看板（各阶段人数统计 + 可视化）
```

## 三、核心角色

### 三·一 角色清单

系统通过 `user.role` 字段区分三种角色，登录后由 JWT token 承载角色信息，后端接口按角色鉴权。

| 角色 | role 值 | 核心职责 | 对应页面 |
|---|---|---|---|
| **候选人** | `candidate` | 浏览职位广场、投递简历、查看我的申请 | 职位广场、职位详情、我的申请 |
| **HR（招聘方）** | `hr` | 发布/管理职位、筛选候选人、写面试备注 | 我的职位、候选人管理、招聘漏斗 |
| **管理员** | `admin` | 全量数据管理、用户管理、系统配置 | 管理后台（P1/P2） |

### 三·二 角色权限矩阵

| 功能 | candidate | hr | admin |
|---|---|---|---|
| 注册/登录 | ✓ | ✓ | ✓ |
| 浏览职位列表 | ✓ | ✓ | ✓ |
| 查看职位详情 | ✓ | ✓ | ✓ |
| 投递简历 | ✓ | — | — |
| 查看我的申请 | ✓ | — | — |
| 发布职位 | — | ✓ | ✓ |
| 编辑/下架职位 | — | ✓（自己发布） | ✓（全部） |
| 查看投递列表 | — | ✓（自己职位） | ✓（全部） |
| 修改申请状态 | — | ✓ | ✓ |
| 写面试备注 | — | ✓ | ✓ |
| 查看招聘漏斗 | — | ✓ | ✓ |
| 用户管理 | — | — | ✓ |

### 三·三 JWT 认证流程

```
用户登录 → 后端校验账号密码 → 签发 JWT (含 userId + role)
    ↓
前端存 token 到 localStorage → axios 拦截器统一加 Authorization Header
    ↓
后端 LoginInterceptor 校验 token → 解析 role → 放行/拒绝
```

### 五·四 接口契约

所有 Controller 统一返回 `Result<T>`：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": { ... }
}
```

| code | 含义 | 触发方 |
|---|---|---|
| 200 | 成功 | 正常返回 |
| 400 | 参数校验失败 | `@Valid` 校验 |
| 401 | 未登录/token 过期 | LoginInterceptor |
| 403 | 无权限 | 角色校验 |
| 404 | 资源不存在 | 查询为空 |
| 500 | 服务器内部错误 | 未捕获异常 |

业务异常码见 `API_DESIGN.md §4`（Phase 3 阶段产出）。

## 七、开发约定

### 七·一 Git Commit 规范

```
type(scope): description

示例:
  chore: 初始化项目骨架
  feat(p4-auth): 添加用户注册接口
  fix(p4-auth): 修复 JWT 过期未刷新
  docs(p1): SRS + 概要设计 + R-01 修复
  feat(p5-LoginPage): 登录表单 + axios 模块
```

**type 清单（8 选 1）**: `feat` / `fix` / `docs` / `style` / `refactor` / `test` / `chore` / `perf`

### 七·二 编码规范

| 维度 | 后端 (Java) | 前端 (Vue 3) |
|---|---|---|
| 缩进 | 2 空格 | 2 空格 |
| 命名 | 类名大驼峰，方法/字段小驼峰 | 组件 PascalCase，文件小驼峰 |
| 注释 | 中文，关键业务逻辑 | 中文，关键业务逻辑 |
| 引号 | 双引号 | 单引号 |
| 分号 | 必须 | 必须 |
| 禁止 | 原生 SQL 拼接、`var` | `var`、混用 UI 库、直接 `axios.get` |

### 七·三 安全红线

- 密码必须 BCrypt 加密，数据库不存明文
- 所有用户输入必须校验（后端 `@Valid` + 前端 el-form rules）
- JWT token 必须由后端 LoginInterceptor 校验
- 前端/配置文件禁止硬编码密钥、密码、token
- 敏感信息（密码、完整 token）禁止打印到日志
- SQL 一律走 MyBatis-Plus 参数化查询，禁止字符串拼接

## 八、测试方案

### 八·一 测试策略总览

本系统采用「测试金字塔」分层策略，从底向上依次为：单元测试 → 接口测试 → 集成测试 → E2E 场景测试。测试重心按 P0 > P1 > P2 优先级递减，确保核心链路 100% 覆盖。

```
         ┌─────────┐
         │  E2E    │  2-3 条核心用户旅程（手动）
         │ 场景测试 │
        ┌┴─────────┴┐
        │  集成测试  │  前后端联调 + 跨模块数据流
       ┌┴───────────┴┐
       │  接口测试    │  ~10 个 P0 接口（Postman/Apifox）
      ┌┴─────────────┴┐
      │   单元测试     │  Service 层 + 工具类（JUnit 5）
      └───────────────┘
```

### 九·六 演示后答疑预案

以下是评委可能的常见提问及回答方向：

| 可能提问 | 回答方向 |
|---|---|
| "为什么用 MyBatis-Plus 而不是 JPA？" | MP 的 LambdaQueryWrapper 防 SQL 注入，跟 MySQL 8.4 配合成熟，中国社区中文文档丰富 |
| "密码加密是怎么做的？" | spring-security-crypto 子模块的 BCryptPasswordEncoder，不引入完整 Security 框架以避开默认 Filter Chain 干扰 |
| "前端为什么用 Pinia 而不是 Vuex？" | Vuex 4 已停止维护，Pinia 是 Vue 官方推荐的下一代状态管理，组合式 API 风格跟 `<script setup>` 统一 |
| "token 过期怎么处理？" | 后端返回 401 → 前端 axios 拦截器统一清 token → 路由守卫重定向 /login，用户无感知 |
| "怎么防止候选人重复投递同一职位？" | application 表 job_id + user_id 联合唯一索引；Service 层投递前先查是否存在 |
| "如果数据量大了分页性能怎么办？" | MySQL LIMIT offset 深度分页问题 —— 可以用游标分页（基于 id > lastId），这是 P2 的优化方向 |
| "接口文档在哪？" | `docs/API_DESIGN.md`（Phase 3 产出，所有接口的请求/响应/错误码完整定义） |
